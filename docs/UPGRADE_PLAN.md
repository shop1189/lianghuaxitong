## 阶段 E — 动态仓位与资金管理（可选 · 按步升级、不急于一步到位）

**背景**：多档止损、加仓/减仓、滚仓、分批止盈等与 **状态机、执行一致性、回测对齐** 强相关；难点不在「写 if」，而在验证与风控责任上移。当前 `evolution_core` 与实验轨以 **单笔、固定 SL/TP、先触发先平** 为主，复杂度可控。

| 子阶段 | 内容 | 何时考虑 |
|--------|------|----------|
| **E0** | **现状基线**：单笔 + 固定档位 + `check_close_trade` 顺序 | 策略优势来自「简单规则 + 大样本」时保持即可 |
| **E1** | **纸面**：状态图、交易所约束、回测能否与实盘对齐 — 先评审再写码 | 确有需要再上 |
| **E2** | **最小闭环**：仅一种动态能力（例：一次加仓后重算 SL，或分批止盈一档） | E1 通过后小流量验证 |
| **E3** | 完整状态机 + **总风险上限** | E2 稳定后再扩展 |
| **E4** | 监控、熔断、分轨策略、上线与回滚 | 与真人团队排期绑定 |

**原则**：若优势主要来自「趋势里滚仓 / 分批止盈」再动动态仓位；若优势来自简单规则与大样本，复杂化可能得不偿失。**每升一级**应有样本或仿真支撑，避免一步到位成黑箱。重大取舍可在 `PROJECT_MEMORY.md` 记一行日期说明。
